# NefroQuest — Estratégia de i18n

## Estado atual (v10.91)

NefroQuest é **PT-first**. O público-alvo são estudantes de medicina e residentes brasileiros.

**Infraestrutura disponível:**

- `window.NQ_CONFIG.lang` — idioma detectado (`'pt'` ou `'en'`), definido em `js/game.js` a partir de `navigator.language`. Default seguro = `pt`.
- `window.t(pt, en)` — helper em `js/utils.js`. Retorna `en` somente se `lang === 'en'` e `en` for string; senão retorna `pt`.

**Onde já há bilíngue:**

- `js/paywall.js` — modal de pricing (usa `isPt ? ... : ...`, pode migrar para `t()`)
- `js/game.js` — constantes `PRICE_MONTHLY`, `PRICE_LIFETIME`

**Onde NÃO há tradução (PT hardcoded):**

- Toda a UI em `index.html` (centenas de strings)
- ~361 template literals em `js/*.js`
- **`data/topics.js`** — 1.003 questões clínicas, 100% PT
- `data/articles.js`, `data/rapid-quiz.js`, `data/refs.js`

---

## Por que i18n completo NÃO é prioridade agora

1. **Conteúdo médico é o maior bloco** — traduzir 1.003 questões de nefrologia é trabalho clínico especializado, não de engenharia. Tradução incorreta de termo médico = risco educacional.
2. **Público é brasileiro** — residência médica BR, título de especialista BR. ROI de inglês é incerto.
3. **Esforço de UI** — centenas de strings espalhadas em HTML + JS sem framework de i18n.

---

## Como usar o `t()` em código novo

```js
// Em vez de:
const msg = isPt ? 'Salvar' : 'Save';

// Use:
const msg = t('Salvar', 'Save');
```

Para strings só-PT (maioria), não precisa fazer nada — ficam como estão.

## Roadmap de i18n (se um dia for prioridade)

### Fase 1 — UI strings (semanas)
- Extrair strings de `index.html` e `js/*.js` para um dicionário `locales/{pt,en}.json`
- Substituir hardcoded por chamadas `t(key)` ou data-i18n attributes
- Considerar adotar uma lib leve (ex: `i18next` standalone) — quebra a regra "sem build step", avaliar trade-off

### Fase 2 — Conteúdo médico (meses, requer médico tradutor)
- Adicionar campo `q_en`, `exp_en`, `opts_en` em cada questão de `topics.js`
- OU manter bancos separados `topics_en.js` carregados conforme `lang`
- Revisão clínica obrigatória da tradução

### Fase 3 — Locale switcher na UI
- Botão de troca de idioma (hoje é automático por `navigator.language`)
- Persistir escolha em localStorage (`nq-lang`)
- `window.NQ_CONFIG.lang` passa a ler do localStorage com fallback pro navegador

---

## Decisão registrada

**Não traduzir em massa por enquanto.** Manter `t()` + `NQ_CONFIG.lang` como fundação para crescimento incremental. Strings novas que tenham versão EN natural (ex: pricing) usam `t()`.
